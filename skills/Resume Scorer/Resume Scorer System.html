Resume Scorer

An automated applicant ranking engine that scores resumes against a Job Description — consistently, transparently, and without manual shortlisting overhead.

◆ Technical Report ◆ 2026 ◆ One Group — HR Operations
01

Introduction

The Resume Scorer automates the first stage of recruitment — screening and ranking applicants against a Job Description (JD). It replaces the slow, subjective process of manually reading every resume with a rule-based scoring engine that applies consistent evaluation criteria to every candidate.

The core principle is straightforward: HR uploads a set of resumes and one JD file. The system reads everything directly, extracts requirements from the JD, matches each resume against those requirements, scores every applicant out of 100, and returns a ranked Excel scorecard.

02

Problem Statement

In resume screening, two competing needs must be balanced:

⏱

Speed

HR teams receive dozens or hundreds of resumes per opening. Manually reading and comparing each one is time-consuming and delays the hiring pipeline.

Bottleneck
⚖

Consistency

Different reviewers apply different standards. A candidate rated "strong" by one screener may be passed over by another. Subjective evaluation leads to inconsistent shortlists.

Risk

Without automation, the screening process is a manual cross-referencing exercise — checking skills, experience, education, and titles across every resume against the JD. The Resume Scorer eliminates this overhead by codifying the evaluation criteria into a deterministic, repeatable scoring model.

03

Input Files

The system accepts two types of uploaded files — one Job Description and one or more resumes. Both PDF and DOCX formats are supported.

File Identification

File Type How It's Detected
Job Description Filename contains "jd", "job description", or "role". Only one JD file is expected per run.
Resumes All remaining .pdf / .docx files are treated as resumes automatically.

Data Extracted from Each Resume

Field Extraction Method
Applicant Name Detected from the first 5 lines of the resume, or a "Name:" label. Falls back to filename if not found.
Skills Keywords extracted from skills sections, summary, and body text
Experience Years of experience and past job titles parsed from work history
Education Degrees, certifications, and fields of study
Industry Domain inferred from employer names and job descriptions
Scanned PDFs: If extracted text is under 100 characters, the resume is flagged as "Unable to extract — possibly scanned" and assigned a score of N/A.
04

JD Parsing & Requirement Tiers

The Job Description is parsed into individual requirements, each assigned a priority tier based on signal words found in the text:

Must-Have
must, required, mandatory, essential, minimum, necessary
Weight: 3×
Should-Have
should, preferred, expected, ideally, strongly preferred
Weight: 2×
Nice-to-Have
optional, nice to have, bonus, plus, desired, advantageous
Weight: 1×
Default tier: If no signal words are found for a requirement, it defaults to Should-Have (weight 2×). This ensures every requirement contributes to the score.

Requirement Clusters

Requirements are grouped into five clusters for transparent, category-level scoring:

Cluster What It Covers
Technical Skills Tools, languages, platforms, software (e.g. MS Excel, ERP, SAP)
Domain Knowledge Industry-specific expertise (e.g. real estate, construction, supply chain)
Experience Years of experience, seniority level, past roles held
Education Degrees, certifications, licenses (e.g. MBA, B.Tech)
Soft Skills Communication, leadership, teamwork, negotiation ability
05

Scoring Model

Each resume is scored out of 100 using four weighted categories. The tier weights from the JD (Must / Should / Nice) are applied on top of these category scores to produce the final result.

46%

Keyword & Skill Match

Overlap of skills, tools, and competencies between the resume and JD requirements.

24%

Experience Match

Years of experience, domain relevance, and alignment with the JD's minimum requirement.

15%

Title / Role Alignment

How closely the candidate's past job titles match the JD role (e.g. "Purchase Manager" vs "Buyer").

15%

Education Match

Degree, field of study, and relevant certifications compared to JD preferences.

Score Bands

Final scores are classified into three bands with conditional formatting in the output report:

🟢
70 – 100
Strong Match
🟡
40 – 69
Partial Match
🔴
0 – 39
Poor Match
06

Processing Logic

The engine processes each scoring run through a six-step pipeline:

1

Locate Uploaded Files

Scan the uploads directory, separate files into one JD and one or more resumes based on filename conventions.

2

Extract JD Text

Read the Job Description file (DOCX via python-docx, PDF via pdftotext) and extract the full text content.

3

Parse JD into Weighted Requirements

Identify each requirement, assign a priority tier (Must / Should / Nice) based on signal words, and group into clusters.

4

Extract Resume Text

Read each resume file, extract the applicant's name from the first few lines, and parse out skills, experience, education, and industry.

5

Score Each Resume

Evaluate every resume across the four scoring categories (Skills 46%, Experience 24%, Title 15%, Education 15%), applying JD tier weights on top.

6

Generate Ranked Report

Output a formatted Excel file sorted by final score (descending), with conditional color formatting and per-category breakdowns.

07

Scoring Formula

The final score per applicant is computed as a weighted sum of four category scores:

Formula
Final Score = (Skill × 0.46) + (Experience × 0.24) + (Title × 0.15) + (Education × 0.15)

Where each category score is itself a weighted match against JD requirements:

Skill Match (46%)
Overlap of keywords, tools, and competencies between resume and JD. Must-have skills carry 3× weight.
Experience (24%)
Years of experience compared to the JD minimum. Domain relevance (e.g. construction industry) provides bonus points.
Example
Skill 72 × 0.46 + Experience 80 × 0.24 + Title 100 × 0.15 + Education 60 × 0.15 = 76.3
Tier weighting: Within the Skill Match category, must-have requirements (weight 3×) contribute proportionally more than nice-to-have requirements (weight 1×). This ensures critical qualifications dominate the score.
08

Edge Cases

The system handles all common edge cases to ensure reliable output:

🔍 Cannot Identify the JD File

⌄

If no file has "jd", "job description", or "role" in its name, the system pauses and asks the user: "Which file is the Job Description?" before proceeding.

📄 Scanned / Image-Only PDF Resume

⌄

If the extracted text from a PDF resume is under 100 characters, it is likely a scanned image. The resume is flagged as "Unable to extract — possibly scanned" and scored as N/A. It still appears in the report for transparency.

🤔 Applicant Name Not Found

⌄

If no name is found in the first five lines of the resume or in a "Name:" label, the system falls back to using the filename (e.g. "resume3.pdf") as the applicant name in the report.

✏️ No Priority Keywords in the JD

⌄

If no signal words like "required", "preferred", or "bonus" are detected, all requirements default to Should-Have (weight 2×). This ensures the scoring model still functions normally.

📂 Duplicate Filenames

⌄

If multiple files share the same name, the system appends a numeric suffix — e.g. resume (2), resume (3) — to avoid confusion in the output report.

🌐 Non-English Resume

⌄

The system attempts to score non-English resumes using keyword matching, but flags the score with a "Low confidence" note in the report cell. HR should review these manually.

📋 Only One File Uploaded

⌄

If only a single file is uploaded, the system asks: "Is this a resume or the JD?" and requests the missing counterpart before proceeding.

10

Conclusion

The Resume Scorer transforms the tedious, subjective process of manual resume screening into an automated, transparent, and consistently applied evaluation workflow. By codifying JD requirements into weighted tiers and scoring resumes across four distinct categories, it removes ambiguity from the shortlisting process.

Every decision the system makes — which skills carry the most weight, how experience is compared, how titles are matched — follows documented rules and produces a per-category breakdown visible in the output report. This transparency builds trust between HR teams and hiring managers.

The result is faster screening cycles, fairer candidate evaluation, and a ranked scorecard that both recruiters and department heads can rely on to make informed hiring decisions.
